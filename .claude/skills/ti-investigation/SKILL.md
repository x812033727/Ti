# Ti 調查任務輸出契約

## 交付物是「結論文字」,不是檔案

你的文字輸出就是唯一交付物,會直接寫回任務看板與教訓庫。**禁止**把結論落檔到 `$TMPDIR`
或任何路徑——別的角色在不同 shell 讀不到,整場會被判失敗(歷史上 9 筆任務因此全滅)。

## 輸出格式(行前綴標記,缺「結論:」=任務未完成)

```
結論: <一段可獨立理解的結論:直接回答任務要查的問題,講清楚答案與根因>
證據: <檔案:行號 或 指令+輸出摘要;可多行,每行以「證據:」開頭>
後續任務: <調查發現需要改碼的具體工作,每行一項、動詞開頭;沒有就不列>
```

特殊出口(擇一,取代上面全部):
- `需人工: <原因>` —— 這件事 AI 做不到(如換發 token、外部後台操作)。
- `需改碼: <原因>` —— 判定必須實際改程式碼才算完成;**不要動手改**,寫這行就好。

## 好結論的標準(會被另一位審稿人試圖推翻)

- 結論直接回答任務問題,不是描述你做了什麼過程。
- 每個關鍵論斷都有對應的「證據:」行,且證據真的支撐論斷(行號對得上主題)。
- 反例(會被打回):「我查了很多檔案,系統很複雜」「結論與證據講的是兩件事」。

## 範例

```
結論: 逾時的根因是 watchdog 只包 run 階段、未涵蓋 fetch,慢速網路下卡在 DNS 解析無人回收。
證據: studio/runner.py:120 watchdog 範圍只到 run(),fetch 在其外
證據: journalctl 顯示 fetch 卡 300s 後才被硬牆逾時殺掉(無中途回收)
後續任務: 把 watchdog 範圍擴到 fetch 階段並補守門測試
```
