# 程式碼審查者 Checklist（長期復用）

- **狀態**：Active
- **類型**：Guide / Process
- **建立日期**：2026-06-10
- **適用對象**：所有擔任程式碼審查者（reviewer）的成員
- **緣起**：沉澱自 no-bare-pytest 方案協作（設計 + 三項實作 + 多輪驗證），抽象為可跨任務復用的審查流程。
- **用法**：每次審查逐項對照；不適用的項目註明「N/A + 原因」，不要靜默跳過。

---

## 0. 審查前（建立判斷基準）

- [ ] 讀懂**驗收標準**：每項驗收能不能轉成一個可執行的「通過/不通過」判斷？模糊就先請工程師補清，別憑感覺審。
- [ ] 確認**任務邊界**：這次審的是哪個任務？哪些屬其他任務／後續待辦？先劃清，避免越界退回或漏審。
- [ ] 掃一遍**既有慣例 / SSOT**：專案有沒有同類既有做法可對齊？新方案是否該沿用而非另起爐灶。

## 1. 設計閘 —— 擋「技術前提是否成立」（最高價值）

- [ ] 方案依賴的**技術前提是否真的成立**？（引擎/版本/平台/相依是否支援所宣稱的用法）
- [ ] 有沒有「看似小事、實則推倒重來」的選型風險？這類問題**必須在動工前**提出。
- [ ] 跨接點**一致性**：同一規則在多處落地（pre-commit / CI / 本機）時，來源是否單一、範圍是否同源？
- [ ] 對工程師的做法給**贊成 / 疑慮 / 替代方案**，務實對齊需求，不空泛。
- 案例：設計稿宣稱「rg + grep fallback」配「lookbehind 排白名單」，但 rg 預設引擎與 grep -E 都不支援 lookbehind ⇒ 設計階段即退回，改兩段式 ERE，省下實作後的引擎返工。

## 2. 實作閘 —— 擋「行為是否正確」

- [ ] **實跑驗證**核心行為，用 exit code / 輸出說話，不靠讀碼臆測。
- [ ] **正反雙樣本**：既驗「該攔的攔到」，也驗「該放的放過」，並備**對照組**證明結果相反（排除「恆綠 / 恆紅」假象）。
- [ ] **主動找反例**，特別是**漏報方向**與高頻邊角（驗收正路通常不覆蓋）。
- [ ] 邊界輸入：多檔 / 空輸入 / 含空白或特殊字元的路徑 / 大小寫 / 不同執行環境（有無某工具）各跑一次。
- 案例：白名單全集跑 exit=0 的同時，必配黑樣本跑 exit=1；額外用對照教學句挖出「同行違規+白名單被整行濾白」的漏報。

## 3. 安全與品質

- [ ] 有無注入面（eval / shell 拼接 / 未引用變數）？輸入是否以 argv 而非字串拼接傳遞？
- [ ] 錯誤訊息**可讀且可定位**（如 `檔名:行號:` 格式），對使用者有指引而非只回非零。
- [ ] 正確性 > 設計 > 風格：抓重點，不為挑剔而退回。

## 4. 結論與交付

- [ ] 每個發現明確分級：**阻擋（退回）/ 非阻擋（待辦）**。
- [ ] 非阻擋缺陷要有**歸宿**：寫進追蹤檔或 issue，不靠口頭。
- [ ] **誠實標注驗證邊界**：能驗到哪講到哪（如環境缺某工具只能模擬），不把「模擬通過」說成「端到端通過」。
- [ ] 產出物（含你自己寫的文件）**符合它所審查的規則**，寫完即驗。
- [ ] 最後一行明確輸出：`決議: 核可` 或 `決議: 退回`；退回時條列必須修正項。

---

## 套用前提（不滿足時先補前提，別硬套流程）

1. **單點規則來源（SSOT）**：規則散落、邏輯重複時，先收斂再審。
2. **低成本可實跑驗證**：行為要能快速驗（秒級 exit code）；需長時間/含隨機/時序的，改用自動化測試套件托底。
3. **驗收標準明確、任務切細**：否則無法逐項給決議。
4. **審查者具備領域基本功**：流程放大專業，但不創造專業；換領域要先確認真的懂那個領域的坑。
5. **只讀驗證、不污染共用狀態**：能建臨時樣本並清理乾淨，不改到共用環境。
